# تصمیمات معماری (ADR) - iFragmentBot

## ۱. انتخاب دیتابیس ترکیبی (Hybrid Database)
- **تصمیم:** استفاده همزمان از MongoDB و SQLite.
- **علت:** MongoDB برای ذخیره‌سازی داده‌های نامتقارن و حجیم (مانند تاریخچه قیمت‌ها برای ۱۰۰ میلیون کاربر) مناسب است. SQLite برای مدیریت سشن‌های تلگرام و تنظیمات سریع بات که نیاز به تاخیر (Latency) بسیار کم دارند، استفاده می‌شود.

## ۲. استفاده از Puppeteer با Stealth Plugin
- **تصمیم:** استخراج داده بصورت Browser-based به جای API مستقیم.
- **علت:** Fragment فاقد API عمومی رسمی و کامل است. برای دریافت داده‌های واقعی که کاربر می‌بیند (مانند قیمت‌های لحظه‌ای حراج)، شبیه‌سازی مرورگر با پلاگین Stealth ضروری است تا از بلاک شدن جلوگیری شود.

## ۳. مدیریت مقیاس‌پذیری (Scalability)
- **تصمیم:** استفاده از `Bottleneck` و `P-Queue`.
- **علت:** برای رسیدن به ۱۰۰ میلیون کاربر، بات نباید با محدودیت‌های Flood تلگرام یا بلاک شدن IP توسط Fragment مواجه شود. این کتابخانه‌ها اجازه می‌دهند نرخ درخواست‌ها را بصورت دقیق کنترل کنیم.

## ۴. رابط کاربری (UI/UX)
- **تصمیم:** استفاده از قالب‌های HTML و تبدیل به تصویر.
- **علت:** نمایش داده‌های پیچیده مالی در فرمت پیام متنی تلگرام محدود است. استفاده از HTML (مانند `marketCard.html`) اجازه می‌دهد کارت‌های گرافیکی زیبا و حرفه‌ای با اطلاعات دقیق به کاربر ارائه شود.
